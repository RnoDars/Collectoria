## Décision Documentée

**Problématique** : Comment implémenter le système d'activités récentes pour le widget du dashboard ?

- Validation rapide du concept
- Pas de nouvelle infrastructure
- Un seul service producteur actuellement
- Refactoring vers Kafka simple plus tard

## Architecture Documentée

### Phase 1 (MVP - Actuelle)

```
Collection Management Service:
  - Toggle Card Possession
  - ↓ (synchrone)
  - Create Activity Record (activities table)
  - ↓
  - Expose via GET /api/v1/activities/recent
```

**Effort** : 2-3 heures  
**Déploiement** : Immédiat

### Phase 2 (Future - Kafka)

```
Collection Service → Kafka Topic → Activity Service → API
    (Producer)       (async)       (Consumer)
                       ↓
                  Notification Service
                       ↓
                  Analytics Service
```

**Effort** : 3-5 jours  
**Déploiement** : Quand déclencheurs atteints

---

## Déclencheurs de Migration vers Kafka

La migration vers Kafka sera déclenchée quand **AU MOINS 2** des conditions suivantes sont remplies :

- [ ] Au moins 2 services différents produisent des événements d'activité
- [ ] Volume d'activités > 10,000 par jour
- [ ] Besoin de notifications en temps réel
- [ ] Besoin d'audit trail complet et durable
- [ ] Problèmes de scalabilité observés
- [ ] Besoin de multiple consumers (Analytics, Notifications)

**Réévaluation** : Tous les 2-3 mois
